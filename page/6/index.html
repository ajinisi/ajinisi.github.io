计算机网络复习之服务

首先，要明确，并非在一个层内完成的全部功能都称为服务。只有哪些能够被高一层实体“看得见”的功能才能称之为“服务”。不同层提供的既有相同的服务，也有不同的服务。
后两篇博客将详述流量控制和拥塞控制

面向连接服务与无连接服务

面向连接的服务按电话系统建模；无连接服务按邮政系统建模

面向连接有三个阶段：连接建立，数据传输和连接释放；无连接没有

面向消息和面向流

不保留消息边界的协议通常称作“基于流的协议”

可靠服务和不可靠服务

现实世界中很多的信道是不可靠的，传输过程中，可能会发生位的错误（由0变为1，或者相反），丢包（例如路由器的缓存溢出），到达的包乱序等等，我们将可靠定义为不错，不丢，不乱。

链路层的可靠性

当链路层协议提供可靠交付服务时，它保证无差错地经链路层移动每个网络层数据损
链路层可靠交付服务通常用于易于产生高差错率的链路，例如无线链路，其目的是本地(也就是在差错发生的链路上)纠正一个差错，而不是通过运输层或应用层协议迫使进行端到端的数据重传。 然而，对于低比特差错的链路， 包括光纤、同轴电缆和许多双绞铜线链路，链路层可靠交付可能会被认为是一种不必要的开销。 由于这个原因，许多有线的链路层协议不提供可靠交付服务

TCP的可靠性

TCP通过序列号、检验和、确认应答信号、（超时）重传机制、连接管理、窗口控制、流量控制、拥塞控制实现可靠性

  • 序列号：是指按照顺序给发送数据包中的每一个字节都标识上一个编号，接收端主机根据接收数据TCP包首部中的序列号和数据长度，来将自己下一步应该接收的序列号作为确认应答返送回去
    • 发现丢包
    • 重新排序，保证顺序
    • 丢弃重复数据
    • 提高效率，可实现多次发送，一次确认
      确认应答处理、重发控制以及重复处理等功能都可以通过序列号来实现
  • （超时）重传机制：当报文发出后在一定的时间内未收到接收方的确认，发送方就会进行重传。TCP含有用于动态估算客户和服务器之间的往返时间RTT的算法，以便它知道等待一个确认需要多少时间
    报文段发出到收到应答中间有一个报文段的往返时间RTT，显然超时重传时间RTO会略大于这个RTT，TCP会根据网络情况动态的计算RTT，即RTO是不断变化的。

    • 在Windows系统中，超时时间都是以0.5s 为单位进行控制的，因此超时时间都是0.5s 的整数倍。不过由于最初的数据包还不知道往返时间，所以最初的重发超时时间一般设置为6s 左右
  • 流量控制，从而确保发送端发送的数据不会使接收缓冲区溢出：总是告知对端在任何时刻它一次能够从对端接收多少字节的数据，这称为通告窗口

计算机组成原理之CPU和MMU

在说中央处理器CPU之前，先要了解计算机存储介质的构成。位于CPU内部的存储介质只有寄存器（CPU内有个寄存器堆，寄存器数量不多，也就20来个），寄存器是容量最小的存储介质，但是是最快的。其他的存储介质都位于CPU外部，容量由小到大分别是cache->内存->硬盘->网络（可视为无限容量）。
在我们运行.out文件中的一条指令时，该指令会在CPU中经历如下步骤：

  1. 由于.out文件保存在磁盘内，由磁盘写入内存，再由内存进入cache区，CPU中的预取器，执行预取指令的工作，把这条二进制形式的指令取进CPU；
  2. 预取器把取到的指令交给译码器，译码器对指令进行分析，获知指令要做的工作，以及完成指令需要使用什么寄存器（如果该指令是完成加法操作，就事先取出eax和ebx寄存器，把加数和被加数放在里面备好）；
  3. 译码器了解具体的运算类型并且准备好寄存器后，有ALU算术逻辑单元完成具体的运算（算逻单元只会+和<<运算，所有运算都可以通过这两个基本运算完成）；
  4. ALU把计算好的结果回写到相应的寄存器，随后再由cache把哪个记录计算结果的寄存器中的内容取走。

CPU中还有一块十分重要的单元——MMU内存管理单元。
MMU的作用：

  1. 虚拟内存与物理内存的映射；
  2. 设置和修改内存访问级别。
    无论我们通过执行一个文件或者系统调用fork()函数产生一个进程，都会产生一片虚拟内存，在32位的系统上，虚拟内存是4G大小（其中0到3G是用户空间可使用的地址，3G到4G是内核空间可以使用的地址）。但是这片内存是虚拟的，并非实际存在的，切实存在的是我们的物理内存，只是我们通过这4G的虚拟内存能够更方便地管理内存空间。由于这块内存是虚拟的，看起来我们的程序相关内容都是存放在这片空间（程序中我们用户所有可以操控的内存地址都是虚拟内存的地址），但是实际还是存放在物理内存上，所以我们就要使用一个方法去把虚拟内存和物理内存之间相互映射，MMU就帮我们完成了这样的映射。在MMU把虚拟内存映射为物理内存的过程中，映射的最小单位是4K 大小（一个Page）,即便虚拟内存有一段区域只有1K大小，在把这块区域映射进物理内存，在物理内存上为它划分空间是时，也会毫不犹豫地划分出4K大小的区域。
    MMU在完成映射的同时，还为CPU设置了对物理内存的访问级别。简单地说，访问级别最高是Ring0，是内核空间Kernel Space；最低是Ring3，是用户空间User Space。
    例如：在程序执行一条命令printf()打印输出一些内容到屏幕上，调用printf()阶段，CPU运行在用户空间，但是printf()还需要向下作系统调用write()，此时CPU进入内核空间，对内存的访问级别发生了改变（由Ring3->Ring0）,这种级别的调整，就是MMU帮助CPU完成的。
    MMU的工作在程序运行起来的时候就开始了，因为从cache上面取来的指令中使用的变量，数值等对应的存储位置就是虚拟内存，MMU需要把它映射到物理内存上，才能真正取出变量中的值。同样的CPU计算完把结果返回给cache也需要MMU协助把物理内存映射回虚拟内存。

设计模式之单例模式

设计模式简单地说，就是把一个问题复杂化，但是复杂化产生的结果，就是使得这个模式能够应用于所有满足该条件的情况。对于C++来说，多态是设计模式的基础。

设计模式简单地分为以下三类：

  1. 创建型模式：常与对象的创建有关，涉及对象实例化的方式；
  2. 行为型模式：常与对象间的通信有关；
  3. 结构型模式：把类与对象组合产生更大的结构。

首先设计模式中实现最简单的，也是十分常用的一种创建型模式：单例模式Singleton。
在socket编程中，我们常常遇到在多线程间，只初始化一次socket资源，建立一次连接，来满足后续通信的需求，这种情况就要使用单例模式。单例模式保证一个类只生成一个实例，在整个程序空间中，这个类仅允许有一个实例对象。

实行单例模式分三步完成：

  1. 类的构造函数私有化；
  2. 用静态方法提供一个该类的全局访问点，使得外部主控函数内可以实现调用；
  3. 定义一个类的静态变量指针，指向本类产生出的变量。

这种方式在调用时才会进行创建，节约了内存空间开销；但同时也有一些缺点，它对于多线程的条件下，是不安全的。例如：有3个线程同时都有创建实例的需求，它们都调用了这个全局访问函数进入了类中进行判断，因为用new创建实例的过程并不是一蹴而就的，并不是个不可中断的原子操作，所以可能出现线程1判断好了，在它之前没有实例产生过，它会进入if逻辑进行创建，在它new到一半创建未完成，还并没有返回的时候，它失去了CPU，CPU的使用权力轮到了线程2手里，线程2也有同样的需求，进入全局GetSingleton函数开始判断，看看在它之前有没有实例产生过，由于线程1的实例创建并没有完成，自然没有一个完整的实例存在于内存中，所以线程2也会认为自己是第一个创建实例的人，进入if逻辑开始new一个实例。这样一来两个线程都产生了各自的实例并且返回，与我们单例的思想相悖（此时有了多个实例）。

为了解决这样的多线程安全问题，我们自然想到了锁机制，在if判断前后加锁，就能保证只有一个线程能够进入if逻辑了（只有抱着锁的那个线程才能执行加锁解锁范围内的语句，其他的线程想要执行只能阻塞等待，直到持锁线程完成工作解锁返回，它才有机会获得锁继续执行）。在锁机制的基础上，我们可以多加一层if判断来更准确的保证单例，维护线程安全，这样，就有了对于多线程友好的double_check单例模式。

但是严格来说，上面的double_check方式也不是完美的，还是由于new的原因，它不是一个原子操作，在调用new的时候，会执行如下三个步骤：

  1. 分配对象空间的内存：memory=allocate();
  2. 初始化对象：ctor_instance(memory);
  3. 设置新对象指向新的内存：_instance=memory;
    在系统编译时，编译器会自动对代码进行优化，导致对于这样不是原子操作的命令进行指令重排，有时会产生内存乱序访问现象，虽然概率不大，但却是十分崩溃的。比如：指令重拍后上面的123步骤重排成了132，由于新的对象还没有初始化就直接指向一块新的内存返回了，会使这个新对象产生不可预知的错误，而且无从查证。解决这个问题最简单的方式就是告诉编译器，这一部分内容我不需要你帮我优化，你就按照我原来写的帮我执行就好了，这样就不会产生指令重排，也就是添加一个volatile关键字(遇到这个关键字声明的变量，编译器对访问该变量的代码就不再进行优化)

操作系统基本概念之并发

并发concurrency和并行parallellism

并发性是独立执行进程的组合；并行性是（可能相关的）计算的同时执行
并发性是指一次处理很多事物；并行性是指同时做很多事情
并行是指两个或者多个事件在同一时刻发生；而并发是指两个或多个事件在同一时间间隔发生
并行是在不同实体上的多个事件，并发是在同一实体上的多个事件。这个实体可以是处理器CPU，或者是多核处理器的一个核
在一台处理器上“同时”处理多个任务为并发，在多台处理器上同时处理多个任务为并行。如hadoop分布式集群。

并发是指程序的 逻辑 结构。非并发的程序就是一根竹竿捅到底，只有一个逻辑控制流，也就是顺序执行的(Sequential)程序，在任何时刻，程序只会处在这个逻辑控制流的某个位置。而如果某个程序有多个独立的逻辑控制流，也就是可以同时处理(deal)多件事情，我们就说这个程序是并发的。这里的“同时”，并不一定要是真正在时钟的某一时刻(那是运行状态而不是逻辑结构)，而是指：如果把这些逻辑控制流画成时序流程图，它们在时间线上是可以重叠的。并行是指程序的 运行状态 。如果一个程序在某一时刻被多个CPU流水线同时进行处理，那么我们就说这个程序是以并行的形式在运行。（严格意义上讲，我们不能说某程序是“并行”的，因为“并行”不是描述程序本身，而是描述程序的运行状态，但这篇小文里就不那么咬文嚼字，以下说到“并行”的时候，就是指代“以并行的形式运行”）显然，并行一定是需要硬件支持的

并发concurrency和并行parallellism的相互联系

应用程序可以是并发的，但不是并行的，这意味着它可以同时处理多个任务，但是没有两个任务在同一时刻执行。应用程序可以是并行的，但不是并发的，这意味着它同时处理多核CPU中的任务的多个子任务。一个应用程序可以即不是并行的，也不是并发的，这意味着它一次一个地处理所有任务。应用程序可以即是并行的也是并发的，这意味着它同时在多核CPU中同时处理多个任务

  1. 并发是并行的必要条件，如果一个程序本身就不是并发的，也就是只有一个逻辑控制流，那么我们不可能让其被并行处理。
  2. 并发不是并行的充分条件，一个并发的程序，如果只被一个CPU流水线进行处理(通过分时)，那么它就不是并行的。
  3. 并发只是更符合现实问题本质的表达方式，并发的最初目的是简化代码逻辑，而不是使程序运行的更快；

并发的实现方式

  1. 显式地定义并触发多个代码片段，也就是逻辑控制流，由应用程序或操作系统对它们进行调度。它们可以是独立无关的，也可以是相互依赖需要交互的，譬如上面提到的素数计算，其实它也是个经典的生产者和消费者的问题：两个逻辑控制流A和B，A产生输出，当有了输出后，B取得A的输出进行处理。线程只是实现并发的其中一个手段，除此之外，运行库或是应用程序本身也有多种手段来实现并发，这是下节的主要内容。、

  2. 隐式地放置多个代码片段，在系统事件发生时触发执行相应的代码片段，也就是事件驱动的方式，譬如某个端口或管道接收到了数据(多路IO的情况下)，再譬如进程接收到了某个信号(signal)。

并行的实现方式

并行可以在四个层面上做到：

  1. 多台机器。自然我们就有了多个CPU流水线，譬如Hadoop集群里的MapReduce任务。
  2. 多CPU。不管是真的多颗CPU还是多核还是超线程，总之我们有了多个CPU流水线。
  3. 单CPU核里的ILP(Instruction-level parallelism)，指令级并行。通过复杂的制造工艺和对指令的解析以及分支预测和乱序执行，现在的CPU可以在单个时钟周期内执行多条指令，从而，即使是非并发的程序，也可能是以并行的形式执行。
  4. 单指令多数据(Single instruction, multiple data. SIMD)，为了多媒体数据的处理，现在的CPU的指令集支持单条指令对多条数据进行操作。其中，1牵涉到分布式处理，包括数据的分布和任务的同步等等，而且是基于网络的。3和4通常是编译器和CPU的开发人员需要考虑的。这里我们说的并行主要针对第2种：单台机器内的多核CPU并行。

并发

并发是一个多义词，意为“同时发生的”。

在后台、服务器领域，并发数是指同一时刻能处理的连接数/请求数，高并发 指同时有很多用户对服务器进行访问；广义上的并发数是指同时使用系统的用户个数，这些用户可能调用不同的 API。严格意义上的并发数是指同时请求同一个 API 的用户个数。本小节所讨论的并发数是严格意义上的并发数。

而在操作系统领域，所谓并发，有三个方面的含义

  1. 多道程序设计，进程被交替执行，表现出一种并发执行的外部特征；
  2. 模块化设计
  3. 操作系统本身

进程间的三种关系

为了简单起见，我们说的并发指的是“进程”的并发执行，在某些系统中，并发的实体是线程而不是进程。

进程之间的感知程度

  1. 进程之间不知道对方的存在，它们之间的关系是竞争；
  2. 进程间接知道对方的存在，它们之间的关系是通过共享进行合作；
  3. 进程直接知道对方的存在（它们有可用的通信原语），它们之间的关系是通过通信进行合作；

进程间的三种控制机制

  • 互斥：它主要源于资源共享，是进程之间的间接制约关系。在多道系统中，每次只允许一个进程访问的资源称为临界资源，进程互斥就是保证每次只有一个进程使用临界资源。如访问控制台、打印机
  • 同步：它主要源于进程合作，是进程间共同完成一项任务时直接发生相互作用的关系。为进程之间的直接制约关系。如生产者-消费者问题，哲学家进餐问题
  • （进程间）通信，IPC：
    同步是一种更为复杂的互斥，而互斥是一种特殊的同步，同步互斥也是一种进程间通信方法

相互竞争的进程，共享资源，需要互斥
共享合作的进程，共享变量，需要互斥和维护数据完整性
通信合作的进程，不共享任何对象，不需要互斥

并发编程和多线程编程

多线程是并发编程的手段之一。当然多线程不一定只有在并发的时候才用到。多线程主要是为了避免一个线程过载影响效率，多线程也可以将单任务截取为多段同时进行，这种情况的多线程并不是为了处理并发。

并发编程的三个概念

原子性：原子性指的是一个或者多个操作，要么全部执行并且在执行的过程中不被其他操作打断，要么就全部都不执行
可见性 ：可见性指多个线程操作一个共享变量时，其中一个线程对变量进行修改后，其他线程可以立即看到修改的结果
有序性，即程序的执行顺序按照代码的先后顺序来执行

目前，WebServer几种主流的并发模型：

  • 多线程，每个线程一次处理一个请求，在当前请求处理完成之前不会接收其它请求；但在高并发环境下，多线程的开销比较大；
  • 基于回调的异步IO，如Nginx服务器使用的epoll模型，这种模式通过事件驱动的方式使用异步IO，使服务器持续运转，但人的思维模式是串行的，大量回调函数会把流程分割，对于问题本身的反应不够自然；
  • 协程，不需要抢占式调度，可以有效提高线程的任务并发性，而避免多线程的缺点；但原生支持协程的语言还很少。

操作系统基本概念之进程线程协程

进程

学习操作系统一上来就会去了解进程的概念。要理解进程的概念，我们可以用对比的方式来说明。即程序、进程、线程和协程的相互区别和联系

  • 程序：就是我们用gcc编译链接好的可执行文件（.out文件），是实际存在的。这些文件保存在磁盘上，并不占用系统资源（系统资源指的是CPU，内存，打开的文件，锁等）。

它是计算机指令的集合，它以文件的形式存储在磁盘上。程序是静态实体（passive Entity），在多道程序系统中，它是不能独立运行的，更不能与其他程序并发执行

  • 进程：活跃的，正在运行的程序（.out文件跑起来就会产生一个进程），是抽象的，动态的概念。进程占用系统资源，在内存中执行。

进程是具有一定独立功能的程序关于某个数据集合上的一次运行活动，是系统进行资源分配和调度的一个独立单位。（包括程序段，相关数据段，和进程控制块PCB）

* 动态性
* 并发性
* 独立性
* 异步性（不确定性）：进程以各自独立的、不可预知的速度向前推进

程序的执行实例被称为进程

* 一个进程就是一个正在执行的程序，包括程序计数器、寄存器和变量的当前值
* 一个具有以下特征的活动单元：一组指令序列的执行，一个当前状态和相关的资源集
  • 线程：线程是进程的一个实体，是CPU调度和分派的基本单位，它是比进程更小的能独立运行的基本单位。线程自己基本上不拥有系统资源,只拥有一点在运行中必不可少的资源(如程序计数器,一组寄存器和栈),但是它可与同属一个进程的其他的线程共享进程所拥有的全部资源

相互比较

  • 程序和进程：同一个程序可以执行多次，产生多个进程，并且即便由同一个程序产生的进程间，也互不影响，彼此隔离。

  • 进程和线程：线程和进程在使用上各有优缺点：线程执行开销小，但不利于资源的管理和保护；而进程正相反。同时，线程适合于在SMP机器上运行，而进程则可以跨机器迁移

  • 协程/微线程，纤程Coroutine：非抢占式的轻量级线程

并发

  • 多道程序
    所有现代的计算机都能同时做几件事情，这都是因为多道程序设计技术使得计算机内存中同时存放几道相互独立的程序，它们在管理程序控制下，相互穿插运行。

早期的CPU都是单道模式，一段时间内只能有一个进程占用CPU，直到这个进程运行结束，退出或者返回，其他正在排队等待执行的进程才能占用CPU进行运算，效率十分低下。为了解决这样的问题，提出了CPU的多道模型。多道实际就是对于CPU的时分复用，CPU划分出非常小的固定长度的时隙片段，同时每个进程也被划分为多个任务片段。CPU先把自己的第一个时隙片段分给进程A使用，一旦这个时隙片段结束，时钟中断到来，进程A无论是否执行完毕，都要主动交出手头的CPU占用权（时钟中断具有强制性）；随后再把下一个CPU时隙片段分给进程B使用，同样的这个时隙片段结束时钟中断到来，进程B也要交出CPU使用权；下一个时隙片段归哪个进程使用是不确定的，谁抢上就是谁的，CPU这样轮询着，直至完成所有进程的运算。这样一来，在某个时间段内，有多个进程都处于启动->运行完成的过程之内，在我们看来，就是多个进程在并列着运行在CPU上。实际上宏观上并行，微观上仍然是串行运行。

  • 多进程
    多道程序是伪并行的，人们很难对多个并行的活动进行跟踪。为此发展出了进程模型，使得并行更容易处理。

  • 多线程
    典型的Unix进程可以看成单线程：一个进程在某一时刻只能做一件事。有了多线程之后。。

    • 通过为每种事件类型分配单独的处理线程，可以简化处理异步事件的代码
    • 多个进程必须使用操作系统提供的复杂机制才能实现内存和文件描述符的共享，而多个线程自动地可以访问
    • 有些问题可以分解从而提高整个程序的吞吐量
      交互的程序同样可以通过使用多线程来改善响应时间

网络上也有人说多道程序对应单CPU，多进程对应多CPU，多线程对应多核CPU。这个说法是不准确的。

PCB

每个进程在它的虚拟内存空间中的内核空间部分（3G到4G部分）都维护着一个进程控制块PCB，它的实质是一个叫做 task_struct 的结构体，里面有许多成员，保存着有关进程的各项信息。其中最重要的几个成员如下：

  • 进程id：每个进程在它正常结束回收或者被杀死回收之前，在系统中都有唯一的非负整数作为它的识别依据，pid_t类型的id号。
  • 进程状态：常见的进程具有5种状态（初始态、就绪态、运行态、挂起（阻塞）态、停止态），当前进程在这一时刻处于何种状态，这样的信息就保存在该结构体中，在程序运行起来后，进程在不同状态间不断切换，这块数据也不断改变。
  • 进程间切换时要保存和恢复的寄存器值：进程A在获得CPU的时间轮片执行了一定时间，有了一些数据的改动或者积累，但是此时时钟中断到来，A主动挂起，失去了CPU，为了防止它在运行期间产生的运算结果丢失（因为下一个获得时间轮片的进程会按照自己的需求重写寄存器，覆盖掉A的结果），PCB中的该成员为当前进程A 记录下它挂起前寄存器中的数据。待到下一个时间轮片被A 获得，就根据这段信息，从上次挂起时的位置继续向下执行。
  • 描述虚拟地址空间的信息：这根指针指向了一个页目录（可视为一个指针表），页目录中的指针再经过层层映射，使得虚拟地址成功有序的映射到物理地址。
  • 描述控制终端的信息。
  • 该进程当前的工作目录。
  • 文件描述符表：PCB中有一个特定的指针，指向该文教描述符表的位置（文件描述符表相当于一个指针数组，数组中的每个指针都指向一个打开的文件的描述信息，这些描述信息包含在一个file struct结构体内，文件描述符表中的每个指针指向这样一个结构体，我们在实际使用文件描述符fd的时候，为了简化操作，都是用文件描述符表中每个指针的下标->一个int数来进行操作，因为用这些下标也都能找到唯一对应的file struct结构体，获取我们想要的已打开的文件的信息）。
  • 进程的可用资源上限（Linux下通过ulimit –a命令也可以查看）。
    等等等…实际的PCB中包含的成员上百项，我们只说了这几种最常见最重要的。

  • 进程共享
    在父进程fork()出来一个子进程后，并不是把父进程的0-3G的用户地址空间简单的copy一份，全部再映射一份进物理内存，而是对于物理内存，遵循读时共享写时复制的原则（仅仅针对有父子关系的进程），是为了节约物理内存。也就是在创建子进程时候，MMU会分析，子进程后续的代码是读操作还是写操作，如果有写操作，比如说，子进程要修改某个全局变量的值，它就会把全局变量所在的.data段从父进程复制一份，映射进子进程独有的一块物理内存；如果后续的操作是读操作，比如说读取某个全局变量的值，那么此时，这个父进程的.data就不需要子进程再复制一份后重新映射到新的一块物理内存，而是子进程和父进程合用父进程的.data段映射的物理内存区域。
    如上所述，要是想通过一个全局变量来在父子进程之间传递信息是行不通的。比如是子进程修改了全局变量值，希望父进程能读到这个修改，是不行的，父进程读不到这个修改，父进程读到的只能是自己对于这个全局变量的操作结果，因为子进程有了写操作，从父进程中隔离出去了，它的全局变量所在物理内存已经和父进程扯不上关系，父进程无法找到子进程新的这段物理内存，自然也读不到子进程实行写操作产生的改变。
    上面所说的读时共享实际上就是共享了以下内容：
  1. 文件描述符表（父子进程可以操作同一个打开的文件）；
  2. mmap建立的映射区（映射区就是物理内存在虚拟地址中映射过来的形式）。
  • 回收进程
    孤儿进程：父进程先于子进程结束，父进程退出后，子进程还在运行，子进程成为孤儿进程，但是一般系统中的init（原始进程）会领养孤儿进程，进行回收。
    父进程先结束，子进程再结束。
    僵尸进程：子进程先于父进程终止（return，exit或者非正常终止）了之后，父进程并未回收已经死掉的子进程，使得子进程残留的资源还在内核中（子进程死掉的时候它的0-3G地址空间会被释放，但是它的PCB仍然保留着放在内核空间中），父进程不回收已经结束的子进程的资源使得子进程成为僵尸进程。
    子进程先结束，父进程再结束。

为了解决僵尸进程，父进程要调用wait()函数或者waitpid()函数回收已经结束的子进程（进程的回收只能由该进程对应的父进程执行，其他进程都没有资格回收）。
在wait()函数调用期间，父进程会阻塞在wait()函数调用的地方，不会继续向下执行，等待子进程结束，回收它的残留资源并且返回它回收成功的这个子进程的pid，在wait()得到返回，回收工作完成时，父进程才会结束阻塞，继续向下执行自己的工作。
对于waitpid()函数，也可以回收调用它的那个进程的子进程，只是他可以选择用非阻塞的方式，采用轮询制度进行回收，再用非阻塞的方式时，父进程仍然可以继续向下执行，而不是停在函数调用处等待回收完成，调用了非阻塞形式的waitpid()函数后，父进程会采用用轮询的方式，每隔一定时间，检查一下子进程是否已经结束，如果结束就回收并对waitpid()进行返回，如果子进程没有结束，就先执行自己的工作，下一个轮询再来检查并回收。

Git的使用之多人合作一

很早就开始使用git了，但是都是一个人写，提交，推送，没有与人合作过。现在在一台电脑上模拟下该过程，即我们要在一台电脑配置多个账号，而这是通过SSH keys实现的。

SSH keys

首先是SSH，而不是SHH，“空眼”患者要注意了。什么是SSH，github有两种推送代码的方式，要么用户名和密码，要么使用SSH keys。为了省去每次访问都要输入用户名和密码的麻烦，采用SSH公钥，密钥对即SSH keys来验证你是否为合法用户。

怎么使用SSH keys

在你的电脑生成了一个唯一的ssh公钥和私钥，公钥放到github上面，当你推送的时候，git就会匹配你的私钥是否跟github上面的公钥是配对的，正确就认为你是合法的，允许推送。现在来确认下你是否已经拥有SSH keys。 默认情况下，用户的 SSH 密钥存储在其 ~/.ssh 目录下。 Windows10系统下，用户的 SSH 密钥存储在其 C:\Users\yourusername.ssh 目录下。

id_rsa就是我们的私钥，id_rsa.pub就是我们的公钥。

  • 如果你没有keys（或者根本没有 .ssh 目录），又或者你不想使用已有的keys，则你可以创建一个新的SSH keys。在一台电脑上使用两个不同的账号，就是采取的此方法建立两个不同的keys。详情见下。

  • 如果你有keys并且想用它连接到Github，则在ssh-agent和Github账户中添加它

新建SSH keys

  1. 使用你的邮箱作为标签，创建一个新的SSH key
  2. 输入文件名，用于保存刚刚生成的 SSH key 代码。不输入将使用 /c/Users/you/.ssh/id_rsa 作为默认文件（如果你是在 ~/.ssh/ 路径下操作的话）。
  3. 输入密码，这个密码会在你提交项目时使用，如果为空的话推送项目时则不用输入。这个设置是防止同一台电脑上的别人私自往你的项目里推送内容。
  4. 因为默认只读取id_rsa，为了让SSH识别新产生的私钥，需将其添加的ssh-agent中，这里要确保你的ssh-agent正在运行
  5. 创建并配置 config 文件，还没有结束，如果现在使用 $ ssh -T git@github.com 只会“欢迎”一个用户。我们还需要添加一个配置文件。
