gcc的学习

太依赖IDE可能不会使得我们对程序的运行有太深刻的认识，学习gcc可以让我们更好地理解操作系统和C语言。

gcc编译链接过程

当我们拿到一个程序源文件hello.c文件，怎么才能将它转化为Linux操作系统可以执行的文件呢？具体的编译链接过程有以下几步：

  1. gcc –E命令：帮我们调用预处理器cpp，预处理器对.c文件做一些简单的预处理工作，例如头文件的展开，程序中宏的替换，删去注释等，预处理器执行完，hello.c文件由原来的源文件编程hello.i的c文件；
  2. gcc –S命令：帮助我们调用编译器gcc，使得c文件变为汇编文件hello.s，完成编译工作，即完成了高级语言向低级语言的转化（主要包括词法分析、语义分析、语法分析产生语法树、源代码的优化、目标代码的生成和优化工作），编译工作也是整个编译链接过程中最复杂，最耗时的一步；
  3. gcc –c命令：帮助我们调用汇编器as，将汇编文件hello.s转换为二进制文件hello.o；
  4. gcc命令：帮助我们调用链接器ld，将动态库或者静态库的相应代码组合入目标文件，从而生成计算机可执行的hello.out文件，随后我们执行这个.out文件，就可以得到hello.c源文件中的执行结果。

以上4步通过gcc hello.c(源文件名) –o(指示生成的文件名的参数) hello.out(生成多个可执行文件名)命令一步即可完成。

gcc中常用的操作

  1. 指定头文件的路径
    在当前编译过程中，如果头文件.h文件与源文件.c文件不在同一个目录下，编译会报
    无法进行，这时我们可以通过指定头文件的目录进行处理，如下：
    gcc hello.c -I ./include –o hello.out
    其中的./include就是我们指定的有文件目录名；I（大写的i）是参数；
  2. 编译时指定宏名：
    在这里指定的宏就不用在每个.c文件中一条条定义了，简化了代码的编写，命令如下：
    gcc hello.c -o hello.out -D DEBUG(宏名)
  3. 选择编译时的优化处理级别：
    gcc hello.c -o hello.out -O2
    -O0，-O1，-O2，-O3；四种级别可选0是不优化，3是最高级优化;
  4. 输出告警信息或者增加调试信息（以供gdb调试时使用）：
    添加参数-Wall可以在编译时输出告警信息，参数-g会为生成的可执行文件增加调试信息，同时生成的这个.out文件体积会变大。

makefile

首先，makefile文件是项目的源代码管理工具，将代码需要编译的命令以一定格式写入makefile，在用make命令执行makefile文件时，就实现了相应的编译。
命名：makefile文件只有两种名字->makefile或者是Makefile；
规则：makefile文件中的语句都有一定的规则，如下
目标：依赖条件
（TAB缩进）命令
目标就是生成的可执行文件的名字；依赖就是生成这个可执行文件所需要的原材料的文件名；命令就是我们通过以来条件生成目标具体要使用的命令。
例如：
app:main.c add.c sub.c mul.c
gcc main.c add.c sub.c mul.c -o app
执行了这样的makefile文件后，就会生成终极可执行目标文件app。这样的编写方式，每次make都会编译所有的.c文件，如果对其中的一个.c文件修改要重新编译，也会编译所有，十分耗时，开销大。

还有一种方式，分模块编译，每次make编译器会对比.c文件最新修改的时间，只编译最新改动的.c文件，示例如下：

app:main.o add.o sub.o mul.o
gcc main.o add.o sub.o mul.o -o app
//最上方的这条规则是用来生成终极目标的，会最后被执行完成，它开始执行的时候发现原材料.o文件不存在，会自动查验下面的子规则，发现自规则生成好了.o文件，才取上来使用，直至凑齐所有的.o文件，生产个终极目标app可执行文件

//子规则
main:main.c
gcc -c main.c
add:add.c
gcc -c add.c
sub:sub.c
gcc -c sub.c
mul:mul.c
gcc -c mul.c
因为每个模块的子规则模式都相似，为了减少冗余，可以使用自动匹配的公式一样的模式规则来适配子规则，以上的子规则语句可以统一用一句模式规则替代，如下：
%.c:%.c
gcc -c $< -o $@
%类似于通配符；$<是规则中的依赖条件；$@是规则中的目标（$符号是取值的含义）。

同样的，makefile中也有一些可以使用的函数，也可以定义变量（但是这里的变量不需要指定任何类型）例如：
变量的使用：
obj= main.o add.o sub.o mul.o
在下面使用到main.o add.o sub.o mul.o的时候就可以用$(obj)来代替。
两个函数：

  1. wildcard函数的使用
    src=$(wildcard ./*.c)返回指定目录下的所有.c文件给变量src（我这里用的是./也就是当前目录的）；
  2. partsubst函数的使用
    obj=$(partsubst ./*.c,./%.o,$(src))将指定目录下所有的.c文件匹配替换成.o文件，并把替换好的.o文件给变量obj（$(src)从这个变量里取出.c文件，并且我这里使用的目录都是当前目录./）

.o文件使用完之后的删除操作：
在makefile的最后生成clean目标，具体操作如下：
clean：
-rm $(obj) app
在执行make clean命令时可以删除obj内包括的所有.o文件和终极目标可执行文件app。

静态库与动态库

工程中我们常常将代码打包成库发给第三方，打包成库可以控制我们想让第三方见到的部分，和不希望让人看到的我们自己核心技术代码的部分。
 静态库
制作步骤：

  1. 用gcc生成对应的.o文件；
    gcc *.c -c
  2. 用ar工具将所有.o文件打包成静态库；
    ar rcs libmytest.a *.o
    libmytest.a是生成的静态库名字，一定要有lib前缀和.a类型。
  3. 发布给第三方测试。
    使用：
    把自己的测试文件main.c编译进静态库测试功能：

-L参数后指定静态库所在目录；
-l参数后指定静态库的库名（去掉lib前缀和.a文件类型后剩下的部分）。
可执行文件中只是打包了main.c测试文件中使用到的.o文件，并不盲目打包整个静态库，如果测试部分不用某个.o文件对应的功能，就不打包这个.o文件（.o文件是最小的打包单位）。

查看：
nm命令可以查看静态库的内容或者查看编译后的可执行文件：
nm libmyCale.a
nm app

  • 静态库的优缺点：
    优点：在向外发布应用程序时，不用提供这个静态库（因为要使用的.o文件已经被打包进了可执行程序内）；库的加载速度快。
    缺点：可执行的应用程序在使用很多静态库时候，或者实际使用到了静态库的所有.o文件的功能，就会使得可执行程序体积非常大；在静态库功能调整代码有变时，整个程序要重新编译，把新的静态库编译进来，非常耗时。

  • 动态库（共享库）
    制作步骤：

  1. 生成与位置无关的.o文件；
    gcc -FpIC -c *.c
    FpIC参数使得代码被放置在程序执行时分配的虚拟内存中一块叫做共享库的代码区域（动态库在共享库内的地址在程序每次执行都是不同的，是动态的，相对的，而静态库每次都是把代码放置在虚拟内存的.test段，地址是固定地，所以我们称静态库生成的.o文件都是与位置有关的，而动态库的.o文件是与地址无关的）。
  2. 将.o文件打包成共享库；
    gcc -shared -o libmyCale.so *.o
    shared参数指定该库是共享库；
    libmyCale.so是新生成的动态库的名称（与静态库命名相似）。

  3. 发布给第三方测试。

    使用：
    gcc main.c -Iinclude -L ./lib -l myCale -o app
    其中Iinclude的第一个字母是I参数，后面跟着动态库的头文件所在目录；
    其他参数解释与静态库一样。
    在使用一个新的动态库时候，常常会出现无法动态连接的情况，因为动态连接器调不起来我们新创建的这个动态库，动态连接器的本质也是一个动态库，它是通过环境变量进行查找连接的，他有自己默认的环境变量值。可以通过修改动态连接器的配置文件（通常在/etc目录下，一个叫做ld.so.conf的文件）把我们要使用的动态库的绝对路径加入到它的环境变量中。

  • 动态库的优缺点：
    优点：由于动态库在实际使用时，没有实际把.o文件打包进可执行文件app中，只是在可执行文件中做了一些标记。在程序真正执行起来前，动态库都不会被加载，只有执行时调用到相应功能时候，才会加载对应模块。所以执行程序的体积小；并且在动态库做更新的时候，不需要重新编译整个程序。
    缺点：加载的速度相对较慢；并且向外发布应用程序时要把动态库一并发布，因为应用程序执行时才把动态库加载进来。

设计模式之单例模式

设计模式简单地说，就是把一个问题复杂化，但是复杂化产生的结果，就是使得这个模式能够应用于所有满足该条件的情况。对于C++来说，多态是设计模式的基础。

设计模式简单地分为以下三类：

  1. 创建型模式：常与对象的创建有关，涉及对象实例化的方式；
  2. 行为型模式：常与对象间的通信有关；
  3. 结构型模式：把类与对象组合产生更大的结构。

首先设计模式中实现最简单的，也是十分常用的一种创建型模式->单例模式Singleton。
在socket编程中，我们常常遇到在多线程间，只初始化一次socket资源，建立一次连接，来满足后续通信的需求，这种情况就要使用单例模式。单例模式保证一个类只生成一个实例，在整个程序空间中，这个类仅允许有一个实例对象。

实行单例模式分三步完成：

  1. 类的构造函数私有化；
  2. 用静态方法提供一个该类的全局访问点，使得外部主控函数内可以实现调用；
  3. 定义一个类的静态变量指针，指向本类产生出的变量。

下面按照这三个步骤实现一个简单的单例模型：

在使用时Singleton single=Singleton::GetSingleton();做这样的静态函数调用，就会进入类中的 static Singleton GetSingleton()进行判断，如果该类的实例从未被创建过，就会创建一个实例并返回。

这种方式在调用时才会进行创建，节约了内存空间开销；但同时也有一些缺点，它对于多线程的条件下，是不安全的。例如：有3个线程同时都有创建实例的需求，它们都调用了这个全局访问函数进入了类中进行判断，因为用new创建实例的过程并不是一蹴而就的，并不是个不可中断的原子操作，所以可能出现线程1判断好了，在它之前没有实例产生过，它会进入if逻辑进行创建，在它new到一半创建未完成，还并没有返回的时候，它失去了CPU，CPU的使用权力轮到了线程2手里，线程2也有同样的需求，进入全局GetSingleton函数开始判断，看看在它之前有没有实例产生过，由于线程1的实例创建并没有完成，自然没有一个完整的实例存在于内存中，所以线程2也会认为自己是第一个创建实例的人，进入if逻辑开始new一个实例。这样一来两个线程都产生了各自的实例并且返回，与我们单例的思想相悖（此时有了多个实例）。

为了解决这样的多线程安全问题，我们自然想到了锁机制，在if判断前后加锁，就能保证只有一个线程能够进入if逻辑了（只有抱着锁的那个线程才能执行加锁解锁范围内的语句，其他的线程想要执行只能阻塞等待，直到持锁线程完成工作解锁返回，它才有机会获得锁继续执行）。在锁机制的基础上，我们可以多加一层if判断来更准确的保证单例，维护线程安全，这样，就有了对于多线程友好的double_check单例模式。

但是严格来说，上面的double_check方式也不是完美的，还是由于new的原因，它不是一个原子操作，在调用new的时候，会执行如下三个步骤：

  1. 分配对象空间的内存：memory=allocate();
  2. 初始化对象：ctorInstance(memory);
  3. 设置新对象指向新的内存：instance=memory;
    在系统编译时，编译器会自动对代码进行优化，导致对于这样不是原子操作的命令进行指令重排，有时会产生内存乱序访问现象，虽然概率不大，但却是十分崩溃的。比如：指令重拍后上面的123步骤重排成了132，由于新的对象还没有初始化就直接指向一块新的内存返回了，会使这个新对象产生不可预知的错误，而且无从查证。解决这个问题最简单的方式就是告诉编译器，这一部分内容我不需要你帮我优化，你就按照我原来写的帮我执行就好了，这样就不会产生指令重排，也就是添加一个volatile关键字(遇到这个关键字声明的变量，编译器对访问该变量的代码就不再进行优化)->stastic Singleton * volatile instance。

操作系统基本概念之并发

并发是一个多义词，它的英文单词是 concurrent ，意为“同时发生的”。在后台、服务器领域，并发是指同一时刻能处理的连接数/请求数，高并发指同时有很多用户对服务器进行访问；

指标

吞吐率，单位时间里服务器处理的最大请求数，单位req/s
从服务器角度，实际并发用户数的可以理解为服务器当前维护的代表不同用户的文件描述符总数，也就是并发连接数。服务器一般会限制同时服务的最多用户数，比如apache的MaxClents参数。

提高并发能力的方法

提高CPU并发计算能力
考虑减少内存分配和释放
考虑使用持久连接
改进I/O 模型

而在操作系统领域，所谓并发，有三个方面的含义

  1. 多道程序设计，进程被交替执行，表现出一种并发执行的外部特征；
  2. 模块化设计
  3. 操作系统本身

进程间的三种关系

为了简单起见，我们说的并发指的是“进程”的并发执行，在某些系统中，并发的实体是线程而不是进程。

进程之间的感知程度

  1. 进程之间不知道对方的存在，它们之间的关系是竞争；
  2. 进程间接知道对方的存在，它们之间的关系是通过共享进行合作；
  3. 进程直接知道对方的存在（它们有可用的通信原语），它们之间的关系是通过通信进行合作；

进程间的三种控制机制

  • 互斥：它主要源于资源共享，是进程之间的间接制约关系。在多道系统中，每次只允许一个进程访问的资源称为临界资源，进程互斥就是保证每次只有一个进程使用临界资源。如访问控制台、打印机
  • 同步：它主要源于进程合作，是进程间共同完成一项任务时直接发生相互作用的关系。为进程之间的直接制约关系。如生产者-消费者问题，哲学家进餐问题
  • （进程间）通信，IPC：
    同步是一种更为复杂的互斥，而互斥是一种特殊的同步，同步互斥也是一种进程间通信方法

相互竞争的进程，共享资源，需要互斥
共享合作的进程，共享变量，需要互斥和维护数据完整性
通信合作的进程，不共享任何对象，不需要互斥

操作系统基本概念之进程

进程

学习操作系统一上来就会去了解进程的概念。要理解进程的概念，我们可以用对比的方式来说明。即程序、进程、线程和协程的相互区别和联系

  • 程序：就是我们用gcc编译链接好的可执行文件（.out文件），是实际存在的。这些文件保存在磁盘上，并不占用系统资源（系统资源指的是CPU，内存，打开的文件，锁等）。

  • 进程：活跃的，正在运行的程序（.out文件跑起来就会产生一个进程），是抽象的，动态的概念。进程占用系统资源，在内存中执行。

  • 程序和进程：同一个程序可以执行多次，产生多个进程，并且即便由同一个程序产生的进程间，也互不影响，彼此隔离。

  • 线程

  • 协程/微线程，纤程Coroutine：非抢占式的轻量级线程

  • 多道程序
    所有现代的计算机都能同时做几件事情，这都是因为多道程序设计技术使得计算机内存中同时存放几道相互独立的程序，它们在管理程序控制下，相互穿插运行。

早期的CPU都是单道模式，一段时间内只能有一个进程占用CPU，直到这个进程运行结束，退出或者返回，其他正在排队等待执行的进程才能占用CPU进行运算，效率十分低下。为了解决这样的问题，提出了CPU的多道模型。多道实际就是对于CPU的时分复用，CPU划分出非常小的固定长度的时隙片段，同时每个进程也被划分为多个任务片段。CPU先把自己的第一个时隙片段分给进程A使用，一旦这个时隙片段结束，时钟中断到来，进程A无论是否执行完毕，都要主动交出手头的CPU占用权（时钟中断具有强制性）；随后再把下一个CPU时隙片段分给进程B使用，同样的这个时隙片段结束时钟中断到来，进程B也要交出CPU使用权；下一个时隙片段归哪个进程使用是不确定的，谁抢上就是谁的，CPU这样轮询着，直至完成所有进程的运算。这样一来，在某个时间段内，有多个进程都处于启动->运行完成的过程之内，在我们看来，就是多个进程在并列着运行在CPU上。实际上宏观上并行，微观上仍然是串行运行。

  • 多进程
    多道程序是伪并行的，人们很难对多个并行的活动进行跟踪。为此发展出了进程模型，使得并行更容易处理。

  • 多线程

网络上也有人说多道程序对应单CPU，多进程对应多CPU，多线程对应多核CPU。这个说法是不准确的。

PCB

每个进程在它的虚拟内存空间中的内核空间部分（3G到4G部分）都维护着一个进程控制块PCB，它的实质是一个叫做 task_struct 的结构体，里面有许多成员，保存着有关进程的各项信息。其中最重要的几个成员如下：

  • 进程id：每个进程在它正常结束回收或者被杀死回收之前，在系统中都有唯一的非负整数作为它的识别依据，pid_t类型的id号。
  • 进程状态：常见的进程具有5种状态（初始态、就绪态、运行态、挂起（阻塞）态、停止态），当前进程在这一时刻处于何种状态，这样的信息就保存在该结构体中，在程序运行起来后，进程在不同状态间不断切换，这块数据也不断改变。
  • 进程间切换时要保存和恢复的寄存器值：进程A在获得CPU的时间轮片执行了一定时间，有了一些数据的改动或者积累，但是此时时钟中断到来，A主动挂起，失去了CPU，为了防止它在运行期间产生的运算结果丢失（因为下一个获得时间轮片的进程会按照自己的需求重写寄存器，覆盖掉A的结果），PCB中的该成员为当前进程A 记录下它挂起前寄存器中的数据。待到下一个时间轮片被A 获得，就根据这段信息，从上次挂起时的位置继续向下执行。
  • 描述虚拟地址空间的信息：这根指针指向了一个页目录（可视为一个指针表），页目录中的指针再经过层层映射，使得虚拟地址成功有序的映射到物理地址。
  • 描述控制终端的信息。
  • 该进程当前的工作目录。
  • 文件描述符表：PCB中有一个特定的指针，指向该文教描述符表的位置（文件描述符表相当于一个指针数组，数组中的每个指针都指向一个打开的文件的描述信息，这些描述信息包含在一个file struct结构体内，文件描述符表中的每个指针指向这样一个结构体，我们在实际使用文件描述符fd的时候，为了简化操作，都是用文件描述符表中每个指针的下标->一个int数来进行操作，因为用这些下标也都能找到唯一对应的file struct结构体，获取我们想要的已打开的文件的信息）。
  • 进程的可用资源上限（Linux下通过ulimit –a命令也可以查看）。
    等等等…实际的PCB中包含的成员上百项，我们只说了这几种最常见最重要的。

  • 进程共享
    在父进程fork()出来一个子进程后，并不是把父进程的0-3G的用户地址空间简单的copy一份，全部再映射一份进物理内存，而是对于物理内存，遵循读时共享写时复制的原则（仅仅针对有父子关系的进程），是为了节约物理内存。也就是在创建子进程时候，MMU会分析，子进程后续的代码是读操作还是写操作，如果有写操作，比如说，子进程要修改某个全局变量的值，它就会把全局变量所在的.data段从父进程复制一份，映射进子进程独有的一块物理内存；如果后续的操作是读操作，比如说读取某个全局变量的值，那么此时，这个父进程的.data就不需要子进程再复制一份后重新映射到新的一块物理内存，而是子进程和父进程合用父进程的.data段映射的物理内存区域。
    如上所述，要是想通过一个全局变量来在父子进程之间传递信息是行不通的。比如是子进程修改了全局变量值，希望父进程能读到这个修改，是不行的，父进程读不到这个修改，父进程读到的只能是自己对于这个全局变量的操作结果，因为子进程有了写操作，从父进程中隔离出去了，它的全局变量所在物理内存已经和父进程扯不上关系，父进程无法找到子进程新的这段物理内存，自然也读不到子进程实行写操作产生的改变。
    上面所说的读时共享实际上就是共享了以下内容：
  1. 文件描述符表（父子进程可以操作同一个打开的文件）；
  2. mmap建立的映射区（映射区就是物理内存在虚拟地址中映射过来的形式）。
  • 回收进程
    孤儿进程：父进程先于子进程结束，父进程退出后，子进程还在运行，子进程成为孤儿进程，但是一般系统中的init（原始进程）会领养孤儿进程，进行回收。
    父进程先结束，子进程再结束。
    僵尸进程：子进程先于父进程终止（return，exit或者非正常终止）了之后，父进程并未回收已经死掉的子进程，使得子进程残留的资源还在内核中（子进程死掉的时候它的0-3G地址空间会被释放，但是它的PCB仍然保留着放在内核空间中），父进程不回收已经结束的子进程的资源使得子进程成为僵尸进程。
    子进程先结束，父进程再结束。

为了解决僵尸进程，父进程要调用wait()函数或者waitpid()函数回收已经结束的子进程（进程的回收只能由该进程对应的父进程执行，其他进程都没有资格回收）。
在wait()函数调用期间，父进程会阻塞在wait()函数调用的地方，不会继续向下执行，等待子进程结束，回收它的残留资源并且返回它回收成功的这个子进程的pid，在wait()得到返回，回收工作完成时，父进程才会结束阻塞，继续向下执行自己的工作。
对于waitpid()函数，也可以回收调用它的那个进程的子进程，只是他可以选择用非阻塞的方式，采用轮询制度进行回收，再用非阻塞的方式时，父进程仍然可以继续向下执行，而不是停在函数调用处等待回收完成，调用了非阻塞形式的waitpid()函数后，父进程会采用用轮询的方式，每隔一定时间，检查一下子进程是否已经结束，如果结束就回收并对waitpid()进行返回，如果子进程没有结束，就先执行自己的工作，下一个轮询再来检查并回收。

SSH keys

首先是SSH，而不是SHH，“空眼”患者要注意了。什么是SSH，github有两种推送代码的方式，要么用户名和密码，要么使用SSH keys。为了省去每次访问都要输入用户名和密码的麻烦，采用SSH公钥，密钥对即SSH keys来验证你是否为合法用户。

怎么使用SSH keys

在你的电脑生成了一个唯一的ssh公钥和私钥，公钥放到github上面，当你推送的时候，git就会匹配你的私钥是否跟github上面的公钥是配对的，正确就认为你是合法的，允许推送。现在来确认下你是否已经拥有SSH keys。 默认情况下，用户的 SSH 密钥存储在其 ~/.ssh 目录下。 Windows10系统下，用户的 SSH 密钥存储在其 C:\Users\yourusername.ssh 目录下。
